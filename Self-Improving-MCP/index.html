The MCP watches itself being used — and fixes itself

Every Leadbay MCP call ships telemetry to PostHog: the tool used, whether it failed, the user's intent, and explicit frustration. Two skills turn that stream into shipped fixes.

Every MCP call, logged as it really happened

Reconstructed into per-user session traces: what they asked → what ran → what broke → what they did next.

Diagnose why · how · what

①

Why are users frustrated?

Cluster friction + failed sessions into themes — each tied to a verbatim quote or error trace.

②

How do we fix it?

Confidence-ranked fixes, split code-verifiable vs needs-investigation.

③

What do they want?

Features inferred only from ≥2 distinct users — each tagged composite vs granular.

An adversarial verify pass tries to refute every claim — uncited findings get dropped before they reach the report.

What /mcp-analysis actually produced

Live run · 856 calls · 15 users · adversarially verified · prioritised into P1/P2/P3 fixes & feature signals.

A finding becomes a fix — --improve

Pick the top finding, and the skill builds it — plan-first, gated, never straight to code.

1Plan (read-only)Investigate root cause → root cause, files to change, eval scenario, risk
→
2ApproveShow the plan; no code is written until it's approved
→
3Build (isolated)Worktree branch · add the eval scenario first · apply the fix

Before-fix eval must FAIL (proving the scenario reproduces the bug) — only then does a fix count.
