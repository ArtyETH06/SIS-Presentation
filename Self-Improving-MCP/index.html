Self-Improving MCP

Based on Real User Experience

THE PROBLEM

Static tools don't learn

⛔

Frozen behaviour

An MCP server ships once and behaves the same forever — regardless of how people actually use it.

🔁

Repeated friction

Users hit the same rough edges again and again. The tool never notices the pattern.

🕳️

Lost signal

Every correction, retry and workaround is real feedback — and it evaporates instantly.

THE IDEA

What if the tool watched itself being used?

An MCP server that observes real interactions, detects where it fails the user, and rewrites its own surface — descriptions, schemas, defaults, even new tools.

“The best documentation is written by watching where people get stuck.”
THE LOOP

The self-improvement cycle

1ObserveCapture real calls, args, errors & retries
→
2DiagnoseFind the highest-friction patterns
→
3ProposeDraft a change to the tool surface
→
4VerifyHuman gate · then measure the result
↺ feeds back into Observe
SIGNALS

What "real user experience" looks like

Retries & corrections

Same tool called 3× with tweaked args → the schema or description is unclear.

Error clustering

A validation error that fires for 40% of calls → a default is wrong.

Workarounds

Users chaining two tools to fake a missing one → propose the missing tool.

IN PRACTICE

A proposed self-edit

// observed: 41% of calls failed validation on `limit`
diff tool: "search_records" {
+  limit: { type: "number", default: 25 }   // was required, no default
+  description: "...; limit defaults to 25 if omitted"
}
// result after ship: validation errors 41% → 3%

Every change is gated, versioned, and measured against the signal that triggered it.

WHERE THIS GOES

Tools that get better the more they're used

From static integrations → living software shaped by real experience.
