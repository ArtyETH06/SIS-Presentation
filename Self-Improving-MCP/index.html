THE PROBLEM

Static tools don't learn

⛔

Frozen behaviour

An MCP server ships once and behaves the same forever — regardless of how people actually use it.

🔁

Repeated friction

Users hit the same rough edges again and again. The tool never notices the pattern.

🕳️

Lost signal

Every correction, retry and workaround is real feedback — and it evaporates instantly.

THE IDEA

The MCP watches itself being used — and fixes itself

Every Leadbay MCP call ships telemetry to PostHog: the tool used, whether it failed, the user's intent, and explicit frustration. Two skills turn that stream into shipped fixes.

/mcp-analysis · diagnose + propose /eval · prove it works

CAPTURE · POSTHOG

What we log on every MCP call

🔧

Tool & outcome

mcp tool called — which tool, ok=true/false, error code, duration.

💬

Intent

triggered_by — the user's verbatim prompt, so we know what they were trying to do.

😤

Frustration

mcp friction reported — category, severity, the user's own words.

Reconstructed into per-user session traces: what they asked → what ran → what broke → what they did next.

SKILL · /mcp-analysis

Diagnose why · how · what

①

Why are users frustrated?

Cluster friction + failed sessions into themes — each tied to a verbatim quote or error trace.

②

How do we fix it?

Confidence-ranked fixes, split code-verifiable vs needs-investigation.

③

What do they want?

Features inferred only from ≥2 distinct users — each tagged composite vs granular.

An adversarial verify pass tries to refute every claim — uncited findings get dropped before they reach the report.

SKILL · /eval

Prove the fix actually works

1Live sessionA real Claude run against the live Leadbay API
→
2Invariant checkDid the required tools actually get called?
→
3Blind judgeFresh context scores 4 dimensions, 0–5
→
4ScorecardArchived to mcp-dashboard, diffed over time

Inside --improve: the scenario must FAIL on the broken code first (proving it reproduces the bug), then PASS after the fix — 5/5/5/5 three times running.

IN PRACTICE

One autonomous fix, gated end-to-end

# /mcp-analysis --days 7 --improve
#  → finding: leadbay_pull_leads fails when lens is ambiguous (P1)

eval --workflow 14  # BEFORE the fix, on broken code
  mission_match 2/5 · tool_selection 3/5   → FAIL  (bug reproduced ✓)

# apply root-cause fix → packages/core/src/composite/pull_leads.ts

eval --workflow 14  # AFTER the fix
  mission_match 5/5 · instruction 5/5 · efficiency 5/5 · tools 5/5
  required_calls ✓ · stability 3/3          → PASS

# → draft PR opened · self-reviewed with /review · never auto-merged
